# 基本的なところ

git は`ローカルリポジトリ`と、外部のリポジトリである`リモートリポジトリ`の二つに分けられる。
例えば、このドキュメントを管理しているローカルリポジトリは`.git`だし、リモートリポジトリは下記で確認できる。
```sh
➜ git remote -v
origin  https://github.com/nichijo/nichijo.github.io.git (fetch)
origin  https://github.com/nichijo/nichijo.github.io.git (push)
```

## WorkingTree, HEAD, Index(Stage)

色々と説明する前に、ひとまず下記三つの説明が必要なので、説明をする
- HEAD
- Index(Stage)
- WorkingTree

### HEAD

以下のような状態のリポジトリがある時
```mermaid
gitGraph
    commit id: "commitA"
    commit id: "commitB"
    commit id: "commitC"
    branch "develop"
    commit id: "commitDevA"
    commit id: "commitDevB"
    commit id: "commitDevC"
```

あなたがもし、mainブランチに居た場合、`HEAD`は`commitC`となり。

```sh
git checkout master
```
```mermaid
gitGraph
    commit id: "commitA"
    commit id: "commitB"
    commit id: "commitC" tag: "HEAD"
```

あなたがもし、developブランチに居た場合、`HEAD`は`commitDevC`となる。
```sh
git checkout develop
```
```mermaid
gitGraph
    commit id: "commitA"
    commit id: "commitB"
    commit id: "commitC"
    branch "develop"
    commit id: "commitDevA"
    commit id: "commitDevB"
    commit id: "commitDevC" tag: "HEAD"
```

現在ブランチの先頭を指す参照と覚えておけばよい。
### WorkingTree, Index

`WorkingTree` は、HEADから変更された差分情報のことで、
`Index` は、後述の `git add` でステージングされたファイルのこと。

```mermaid
stateDiagram
    Committed --> HEAD
    HEAD --> WorkingTree: "なにがしかの編集"

    state "コミット済み" as Committed {
        state HEAD {
        state "hoge.txt" as h_hoge
        state "fuga.txt" as h_fuga
        state "piyo.txt" as h_piyo
        }
        note left of HEAD
            ブランチ上の最新ファイル
        end note
    }

    state "未コミット" as UnCommitted {
        direction LR
        WorkingTree --> Index: "git add"

        state WorkingTree {
            state "hoge.txt" as w_hoge
            state "piyo.txt" as w_piyo
        }
        note left of WorkingTree
            HEADからの差分があるファイル
            .git 管理下のファイルを編集することで勝手に登録される
        end note
        state Index {
            state "fuga.txt" as i_fuga
            state "moge.txt" as i_moge
        }
        note left of Index
            WorkingTreeから add(後述) したファイル
        end note
    }
```

## add

`add` は、`WorkingTree`から任意のファイルを`Index(Stage)`に移動する。

下記のようにコマンドを実行する。
```sh
git add hoge.txt
git add fuga.txt
```

```mermaid
stateDiagram
    direction LR
    hoge --> hoge.txt
    fuga --> fuga.txt

    state WorkingTree {
      direction LR
      state "hoge.txt" as hoge
      state "fuga.txt" as fuga
      piyo.txt
    }
    note left of WorkingTree
        hoge.txt, fuga.txt は
        ステージングに移動され、WorkingTree上から
    end note
    state Staging {
      direction LR
      hoge.txt
      fuga.txt
    }
    note right of Staging
        WorkingTreeからStagingに移動された
        hoge.txt fuga.txt は、git commit 時に
        コミット対象となる
    end note
```

## branch

gitには`ブランチ`という機能がある。`ブランチ`は、簡単に言えばブランチごとにソースコードを分けておける機能で、複数の作業を同時進行するときや、お試しで幾つかのパターンで改修を加えるときなんかに使う。

現在のコミットログはこう。現在位置は`HEAD`
```mermaid
gitGraph
    commit id: "0-00a7a51"
    commit id: "1-0769b79"
    commit id: "HEAD" type: HIGHLIGHT
```
あなたは上司から、2件のアプリケーションの改修依頼を受けたとして…  
一つめの改修では `ViewA` と `GodClass` が。  
二つ目の改修では `ViewB` と `GodClass` の改修が必要になったとする

```mermaid
classDiagram
    class GodClass
    class ViewA
    class ViewB

    GodClass <.. ViewA : use
    GodClass <.. ViewB : use
```

こういう場合、作業ごとにブランチを切ったほうが、作業が混沌としないで済む
```sh
git branch work1
git branch work2
```

```mermaid
gitGraph
    commit id: "0-00a7a51"
    commit id: "1-0769b79"
    commit id: "HEAD"
    branch work1
```

作業１を進める。ある程度の作業ごとでコミットすると良いだろう

```mermaid
gitGraph
    commit id: "0-00a7a51"
    commit id: "1-0769b79"
    commit
    branch work1
    commit id:"view1fix" type: HIGHLIGHT
    commit id:"godclass fix" type: HIGHLIGHT
```

ブランチの切り替えには `checkout` を使う。例えば下記は work1 で作業中の状態。
```mermaid
gitGraph
    commit id: "0-00a7a51"
    commit id: "1-0769b79"
    commit
    branch work1
    commit
    commit id:"HEAD" type: HIGHLIGHT
```

チェックアウトをすると、ブランチのHEADへ移動する
移動すると、ファイルの状態もコミットの状態に戻る。
```sh
git checkout master
```
```mermaid
gitGraph
    commit id: "0-00a7a51"
    commit id: "1-0769b79"
    commit id: "HEAD" type: HIGHLIGHT tag: "ココに移動"
    branch work1
    commit
    commit
```

合間合間に作業２を進めても良い
```mermaid
gitGraph
    commit id: "0-00a7a51"
    commit id: "1-0769b79"
    commit
    branch work1
    branch work2
    checkout work1
    commit
    checkout work2
    commit
    checkout work1
    commit
    checkout work2
    commit type: HIGHLIGHT tag:"ココで作業"
```

仮に途中で作業が不要になったりしたらば、ブランチを消せばよい
```sh
git branch -D work2
```
通常 `git branch -d` で消すが、mainブランチにマージしていないとエラーが出て怒られる。
なので `git branch -D` として強制的に削除する。

```mermaid
gitGraph
    commit id: "0-00a7a51"
    commit id: "1-0769b79"
    commit
    branch work1
    branch work2
    checkout work1
    commit
    checkout work2
    commit type: REVERSE tag:"削除"
    checkout work1
    commit
    checkout work2
    commit type: REVERSE tag:"削除"
```

最終的にはこうなる

```mermaid
gitGraph
    commit id: "0-00a7a51"
    commit id: "1-0769b79"
    commit
    branch work1
    checkout work1
    commit
    checkout work1
    commit
```

## rebase

`rebase` は、ブランチを使うにあたって覚えておきたい機能の一つで、主に以下の用途で使われる。
1. 現在のブランチを、main等の親ブランチに追随させたい時
2. 現在のブランチの内容をコミットし直したい時

例えば `work1` に、mainの内容を反映させたいような時は…

```mermaid
gitGraph
    commit id: "commit1"
    commit id: "commit2"
    commit id: "commit3"
    branch work1
    checkout work1
    commit id: "work1-commit1"
    commit id: "work1-commit2"
    checkout main
    commit id: "commit4"
    commit id: "commit5"
```


```sh
# 現在 `work1` ブランチに居る
➜ git branch
  main
* work1
# rebase を実行する
➜ git rebase master
```
とするとこうなる

```mermaid
gitGraph
    commit id: "commit1"
    commit id: "commit2"
    commit id: "commit3"
    commit id: "commit4"
    commit id: "commit5"
    branch work1
    checkout work1
    commit id: "work1-commit1"
    commit id: "work1-commit2"
```

また、 `-i` オプションを使用することで、過去のコミットを奇麗にまとめあげることができる
（他にも、`commit --amend` と組み合わせて編集したり、ファイルを除外したりと色々できるので覚えて損はない）

```sh
# rebase を実行する
➜ git rebase -i master
```

```txt
pick work1-commit1 commit message1
pick work1-commit2 commit message2

# にゃんたらかんたら
```

のような感じで、過去のコミットの一覧が出てくるんで…
頭の「pick」の部分を変更することでそのコミットをまとめることができる

```txt
pick work1-commit1 commit message1   # コミットを採用
squash work1-commit2 commit message2 # 採用するがひとつ前のコミットにまとめる
```

これを実行すると

```mermaid
gitGraph
    commit id: "commit1"
    commit id: "commit2"
    commit id: "commit3"
    commit id: "commit4"
    commit id: "commit5"
    branch work1
    checkout work1
    commit id: "work1-commit1" tag: "work-commit2の含まれている"
```


```
git checkout master
git merge work1
```

```mermaid
gitGraph
    commit id: "commit1"
    commit id: "commit2"
    commit id: "commit3"
    commit id: "commit4"
    commit id: "commit5"
    commit id: "work1-commit1" type: HIGHLIGHT
```

となり、奇麗にマージすることができる。
